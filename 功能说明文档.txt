一、 基础开发信息
系统目标：基于参考文本的风格模仿与小说正文续写，兼具同人小说与原创小说的自动化创作与设定推演能力。
解决痛点：角色设定冲突，上下文遗忘，文风不统一。
两大受众：作者续写（成本敏感型需求）与读者同人（衍生创作需求）。
延伸功能：AI 人物对话或互动、基于原著的同人短篇创作。
核心大模型：DeepSeek V3（标准模型）与 DeepSeek R1（推理模型）（通过 API 接入，基础请求地址为 https://www.google.com/search?q=api.deepseek.com）。
相关工具：FastAPI（用于构建后端服务与 API 路由架构）、Jieba（用于自然语言处理、高频词分词与词性标注）、Trae（用于辅助工作流搭建与代码生成）、Git（用于代码版本记录与储存库管理）。
操作交互：终端环境支持直接右键复制与粘贴操作；系统底层具备物理目录严格映射机制（包含 reference_novels 作为原文输入库、text_style_imitation 作为特征输出库、novel_projects 作为工作台项目库等核心子目录），确保全流程产生的数据与文件实现自动化分类与层级归档。
二、 页面布局简单陈述
系统界面分为左、中、右三大功能区域，具体布局与交互功能如下：
1. 左侧区域（全局控制台）
用于系统全局环境配置与当前对话状态管理。布局自上而下包含四个功能模块：
布局与显示面板：提供右侧小说工作台的显示开关、中侧与右侧区域屏幕宽度的占比调节滑块，以及系统全局文本字号的调节滑块。
API与防护面板：包含 API 密钥配置框、全局禁用词输入框。提供“自动重试次数”与“敏感词容忍度”参数输入，用于控制流式生成触发限制时的 API 重新请求逻辑。
历史对话管理面板：提供新建独立对话功能与一键复制当前对话功能。复制功能可将当前对话的全部设定与历史上下文无损克隆，用于剧情岔路口的多分支推演。下方列表用于展示及切换已有对话记录。
当前对话参数面板：支持对选中的单个对话设定独立变量。包含对话标题输入、处理模型选择（DeepSeek V3 或 R1）、当前对话专属的局部禁用词输入框、大语言模型生成参数调节（控制随机性的 Temperature 与控制采样范围的 Top-P 参数滑块）、系统提示词（System Prompt）输入框，以及一键清空当前对话历史记录的操作按钮。
2. 中侧区域（智能对话区）
用于执行与大语言模型的交互式指令对话及流式文本呈现。布局包含三个部分：
顶部信息栏：展示当前对话名称、运行的模型标识，以及触发局部与全局禁用词的实时拦截统计次数。
消息展示区：用于展示用户指令与大模型输出结果。
在模型进行流式输出时，前端实时执行正则表达式扫描，若触发全局或局部禁用词，直接在文本中调用红底加粗的 CSS 样式将违禁词高亮标出。
消息数据结构底层支持多版本树（Versions Tree），在进行续写迭代时，同一条提示词生成的多个回答版本会折叠在同一层级供用户切换。
当光标悬停于大模型生成的正文消息框时，系统显示“导入小说末尾”操作按钮，点击后通过后台 API 将该段落直接追加至右侧工作台当前选中章节的 .txt 文件末尾，并实时刷新右侧编辑器。
底部输入区：包含文本输入框与发送指令按钮。在大模型进行流式生成期间，发送按钮将切换为强制中断请求的停止按钮。
3. 右侧区域（小说工作台）
包含“内容编辑”与“自动化工作流”两个可点击切换的面板，用于小说项目的具体执行与管理。
内容编辑面板：顶部包含项目选择下拉菜单与章节选择下拉菜单，并提供新建章节与重命名当前章节的交互按钮。下方为正文编辑器，具备实时字数统计展示与防抖自动保存机制（在用户停止输入 1000 毫秒后自动覆盖保存本地文件）。
自动化工作流面板：分为两部分。第一部分为特征提取与统计操作区，提供参考文本源文件选择下拉菜单，并包含触发统计特征与提取风格脚本的执行按钮；第二部分为项目初始化操作区，包含小说书名输入框与一键生成工程结构按钮，执行后将在本地物理目录自动创建标准化的项目文件夹、子目录及预设的空白文档模板。
“知识库管理”标签页：用于解决大语言模型提取信息时产生的数据遗漏或事实幻觉问题；该面板将功能二、功能三及功能四生成的关键词库、角色信息卡、世界观设定等结构化数据进行可视化渲染并转换为表单格式，提供明确的增、删、改、查交互操作，允许用户手动修改与维护各项设定信息，同时在底层工作流中强制系统在执行功能五（小说创作）前，直接读取该面板最新修改并落盘的物理存储数据，从而替代初次提取的缓存数据，确保文本生成阶段的设定准确性。

三、主要功能说明
功能一：特征提取
执行逻辑：分为本地物理指标计算与大模型深层文风提取两个阶段。首先通过本地 NLP 算法脚本处理全量纯文本文件，计算基础物理与语言学统计指标。随后，截取规定长度（50000字内）的原文样本调用大模型 API，提取深层文风特征（包含世界观标签、叙事节奏、语体色彩、叙事语调、描写风格、人称视角、心理呈现途径及修辞手法偏好）。
涉及提示词：“特征提取提示词”（完整见附录一）。
完整统计指标：
统计断句、完整句子、段落的平均长度和方差（标准差）；统计高频词；统计标点符号使用率；统计双引号 “ ” 内的字数，“ ” 的出现频率，以及“ ” 的位置偏好（计算：统计提示语在对话中的位置占比——前置（xx：“…”）、中置（“…”，xx，“…”）、后置（“…”，xx。）、无提示语（直接引用）。）；词性分布比例；重复词汇比例
进阶方向：文风偏移量化监测。在生成后续章节时，对新生成文本执行本地 NLP 特征提取算法。计算其长短句方差、标点符号使用率及对话位置占比，并与原始参考文本的基线数据进行数学距离（如欧氏距离）计算。当偏离值超出预设的安全阈值时，系统将提示是否重试。
功能二：关键词库提取
执行逻辑：基于全量文本进行词汇扫描。首先调用 Jieba 分词处理全量文本，统计有效总词数与独立词汇数，计算对数 TTR 与根号 TTR。系统采用动态推荐算法，最高提取上限为 8000 词。动态参数计算公式为：N=50×log10unique_words×3total_valid_words
且允许用户手动输入具体的正整数（默认 500）来强制指定高频词的提取名次（Top N）。
提取全量高频词后，将其转换为向量形式并在向量数据库中检索对应的原文文本块。将词汇及其关联的上下文文本块一并提交给大模型，调用大模型对词汇进行去重与分类，输出基础的关键词库。
涉及提示词：“关键词库提取提示词”（完整见附录二）。
进阶方向：利用海量ai生成文本提取常规词库，然后把关键词库与准备好的常规词库比对，整理负面词库，即选出在常规词库中存在但极少或未在当前项目关键词库中出现的词汇。在此基础上，进一步引入词频-逆文档频率（TF-IDF）算法与文本排序（TextRank）算法，为提取出的高频词汇与专有名词分配数学权重。在向大语言模型输入词库时附带该权重参数，以此精确约束各个专属词汇在生成文本中的出现频率，防止大语言模型在生成时过度堆砌罕见专有名词。
功能三：专属词库、世界观与角色信息提取
执行逻辑：在功能二生成的关键词库基础上，取消原文 50000 字的截断限制，改用全局向量检索机制。
第一步：提取该文本的专属词库。
第二步：针对关键词库中的核心实体（人物名：包括别名、地名、势力名），将其作为查询条件输入向量数据库，检索并提取全量文本中所有包含该实体的高相关度历史文本块。
第三步：将检索返回的实体相关文本块输入大模型，构建并输出结构化的世界观设定与角色信息卡（涵盖人物别名、专属称呼、势力隶属关系与核心境界/能力设定）。
涉及提示词：“专属词库提取提示词”（完整见附录三）、“世界观整理提示词”（完整见附录四）、“角色信息卡整理提示词”（完整见附录五）。
进阶方向:角色信息卡不再是静态的单一文档，而是具有版本更迭的动态字典。每次功能五生成小说后，利用功能三重新提取该章的人物状态并更新信息卡。记录格式附带章节锚点（例如：年龄：[第一章:10岁] -> [第五十章:12岁] -> [第一百章:15岁]）。在后续提取与检索时，系统自动屏蔽旧数据，严格只取该属性的最新锚点（最终状态）作为提示词的输入条件。
功能四：设定补全与剧情压缩
执行逻辑：处理信息残缺或需拓展的设定架构，并构建分层的检索索引。
一方面，当用户仅提供部分基础设定时，调用大模型基于已有信息进行推演与填充，补全完整的世界观架构与角色信息卡。
另一方面，针对前置连载正文，采用本地算法（正则表达式切分章节 + Jieba 关键词提取）生成章节维度的伪摘要，不消耗 API 额度。将生成的摘要文本进行向量化并存入向量数据库，与其对应的长文本正文块建立映射关联，形成“摘要-正文”分层检索（Hierarchical RAG）结构，释放模型上下文窗口。
涉及提示词：“世界观整理提示词”（完整见附录四）、“角色信息卡整理提示词”（完整见附录五）、“剧情压缩提示词”（完整见附录六）。
进阶方向：提取标准化叙事结构模板。在执行剧情概括与压缩时，通过自然语言理解模型对原著文本的剧情起伏进行结构化拆解。识别并标注出“触发事件”、“冲突升级”、“高潮节点”、“结局收束”等核心叙事单元，形成抽象的剧情结构模板。在进入后续小说创作的原创或同人分支时，将该结构模板作为底层规则进行输入，约束新生成的剧情在宏观发展流程上与原著的叙事框架保持物理一致。
功能五：小说创作
*执行逻辑：
以工作台中预设剧情为基础，结合 RAG 检索机制与前置约束条件进行正文生成。
执行流为：用户选择或输入本章剧情大纲 -> 系统将大纲文本向量化，在向量数据库中执行相似度检索，召回前置章节中高度相关的文本块（如人物历史对话、特定场景细节） -> 调用提示词生成本章详细大纲 -> 依据详细大纲，并将召回的历史文本块作为上下文参考资料注入提示词，开始正文创作。
*该功能包含两个核心创作分支：
同人创作分支：需调用功能一、功能二、功能三的数据，以及 RAG 检索出的原著细节作为基础约束条件。若加入原创人物，则额外叠加功能四的补全数据。
完全原创分支：需调用功能一、功能二、功能四的数据，以及 RAG 检索出的前文生成记录作为约束条件
涉及核心提示词：“本章大纲提示词”（完整见附录七）。
进阶方向：严格利用上下文消息列（Message List）的层级。将本章大纲、相关信息等“主动输出信息”放置于 User Prompt（用户消息）中；将角色卡、世界观中与用户消息无关的信息作为“被动约束信息”全部放置于 System Prompt（系统消息）中，并添加最高权重的否定指令（如：“以下为底层后台逻辑，禁止在正文叙述中直接陈述、概括或罗列这些属性”）。使大模型在底层逻辑上明确指令的区别。
功能六：剧情方向推演
执行逻辑：根据已生成的分章正文和当前世界观整体设定进行推演。推演前，向向量数据库发起针对“未解决事件”、“未使用道具”或“未出场角色”的自动化检索查询，提取处于休眠状态的剧情线索。将现有逻辑链条与检索出的历史线索一并输入大模型，调用大模型输出后续 3 至 5 个可能的剧情发展分支及方向建议，供用户在智能对话区进行判断与选择。
涉及提示词：“剧情推演提示词”（完整见附录八）。
进阶方向：建立未收束剧情线索的自动化追踪与优先级排序机制。剧情推演操作不仅依据最近生成的正文，需进一步通过算法遍历前置剧情摘要数据集，定向提取已记录但未产生后续物理或逻辑影响的实体信息（如未消耗的道具、未确认下落的次要角色）。将上述线索按照章节跨度与预设重要性进行加权计算，并强制转换为剧情推演的初始输入参数，约束大语言模型优先输出能够处理这些早期前置设定的剧情发展分支。
功能七：校验文本
执行逻辑：对功能五生成的正文文本进行自动化交叉验证与事实核查。
首先，提取新生成文本中的核心动作与实体参数进行向量化。其次，在向量数据库中定向检索这些实体与动作的历史设定文本块。最后调用大模型，将新生成的文本与当前项目的世界观设定文档、角色信息卡、本章预设大纲，以及 RAG 检索返回的历史确切记录进行多重比对。检查并标识出生成的文本中是否存在设定冲突、人物行为逻辑崩坏或偏离既定剧情大纲的情况，确保生成质量。
涉及提示词：“文本校验提示词”（完整见附录九）。
四、 技术细节与难点
注意：配置国内镜像网站
已经实现的技术细节：
系统并发与任务调度：特征提取与统计类脚本（如 text_stats.py）设计了 GUI 与静默双重启动模式。后端 FastAPI 调用时采用进程级（subprocess.Popen）异步拉起，确保耗时极长的全量文本分析不会阻塞主线程，保障前端高并发交互的流畅度。
文本编码自动降级与容错：针对多源纯文本小说文件编码混乱问题，底层文件 I/O 设计了降级读取机制。优先以 UTF-8 编码解码，若捕获 UnicodeDecodeError 异常，则自动无缝降级为 GBK 编码进行二次尝试，大幅提升系统对原生 txt 文件的兼容性。
前端 Markdown 渲染与安全映射：大模型返回的流式文本在前端不仅会经过正则拦截器进行敏感词检测，还会通过 marked.js 实时转换为 HTML 标签，从而支持标题、粗体、代码块等富文本语义还原。
边界路由防丢机制：针对用户未明确选中具体章节却执行“导入小说末尾”操作的边界情况，系统 API 层设计了保底路由算法，会自动读取物理目录并默认将生成文本追加至排序第一的物理文件末尾，防止生成的上下文数据因指针丢失而未落盘。

六、代码实现
步骤一：基础环境与目录架构搭建
操作目的：在本地物理硬盘上建立项目的基本框架，隔离运行环境，并安置所有的基础配置文件，为后续的数据读写提供确切的路径。
1.创建虚拟环境：
操作：在 Trae 编辑器底部的终端（Terminal）中，复制并右键粘贴命令 python -m venv venv，按回车执行。
解释：这会在项目中创建一个名为 venv 的独立 Python 环境，防止本项目安装的代码库（如 jieba）与其他项目冲突。

行文结构：篇章内部段落与情节之间的逻辑组织与架构方式等分类：总分总（总分/分总）/承上启下/悬念设置（制造扣子）/伏笔照应（草蛇灰线/首尾呼应）/明暗双线交织等
叙述方式：交代事件发生过程时的时间序列调度等分类：顺叙（按时间先后推进）/倒叙（先交代结局或高潮）/插叙（中途切入补充背景回忆）/补叙（事后补充遗漏细节）/平叙（多线同时发生的交叉剪辑）等
