一、 市场现状与需求洞察

***

1.1 行业宏观市场分析

当前网络文学行业正处于商业模式转型与底层技术变革的交汇期，整体呈现出传统图文业务承压与人工智能新兴赛道爆发并存的宏观态势。具体表现为以下四个物理与经济维度：

1\. 传统网文平台面临盈利与增长阵痛

传统网文平台单纯依赖付费阅读与大IP重度影视化改编的商业模式面临增长瓶颈与长周期风险。以阅文集团（起点中文网母公司）发布的2025年财报数据为例，其全年总营收同比下滑9.3%，净亏损扩大至7.76亿元（主要受旗下新丽传媒18亿元的商誉减值影响）。这表明传统图文内容的直接变现与重资产影视开发的利润空间正在收缩，平台急需通过技术手段寻找第二增长曲线。

2\. 创作者群体收入呈现极端分化

以番茄免费小说为代表的免费阅读模式有效扩大了下沉市场的用户基数，但算法推荐机制导致了严重的流量集中化与产能内卷。以番茄小说等头部免费平台为例，绝大部分底层与中腰部作者的月收入不足3000元，部分甚至长期低于1000元。每日固定更新4000至8000字的高强度劳作与极低的经济回报产生直接冲突。这一生存现状构成了中低收入作者群体对“降本增效”及自动化文本辅助生成工具的绝对刚性需求。

3\. AI 漫剧与微短剧市场蓬勃发展

AI 技术正在重构网文IP的轻量化改编链路。由于传统影视开发周期长、试错成本高，AI 漫剧与微短剧因其极高的资金周转效率成为行业核心增量。数据显示，2025年 AI 漫剧市场规模已达168亿元，并预计在2026年突破240亿元。AI 工具链的介入使漫剧的生产成本平均降低70%以上，生产环节耗时缩短约三分之一。这种技术赋能下的低成本衍生模式，为海量中腰部网文的版权变现提供了物理通道。

4\. 网文出海规模化与多语种生态确立

中国网络文学海外活跃用户规模目前已接近4亿人。2025年，AI 翻译技术在网文出海领域实现了底层规模化应用。以起点国际（WebNovel）为例，全年新增 AI 翻译作品超万部。大语言模型的介入使得多语种翻译成本降低约90%，不仅巩固了北美与东南亚等成熟市场，更直接推动了拉美等新兴市场的爆发式增长。网文出海的商业模式已彻底从高成本的人工“文本输出”阶段，跃迁至“AI 自动翻译+本土化生态共建”的高效扩张阶段。

1.2 目标用户画像 (Persona)

系统针对两类核心受众进行开发：

第一类为小说原作者。其核心需求为正文续写，属于成本敏感型用户。该群体关注大语言模型 API 调用的经济成本与本地计算资源的分配，要求系统在维持稳定输出质量的同时降低长篇连载的资金消耗。

第二类为读者群体。其核心需求为同人衍生小说创作。该群体关注生成文本对原著的设定还原度与风格保真度，要求系统具备精确解析、存储并调用原著角色信息与世界观架构的能力。

1.3 核心痛点与潜在需求挖掘

核心痛点主要包含三项：

其一，角色设定冲突。大语言模型在生成时容易违背已有的人物性格、境界能力或社会关系设定。

其二，上下文遗忘。受限于大模型单次交互的上下文窗口容量上限，在长篇连载中后期，系统极易遗漏早期埋设的伏笔、未消耗的道具或次要角色。

其三，文风不统一。多次分章节生成或不同模型交叉使用，导致生成文本的句段长度、叙事节奏、词汇偏好产生明显断层。

潜在需求挖掘主要包含两项延伸功能：基于已有小说设定体系的人工智能单体角色对话与互动；完全依托原著世界观与角色状态切片的独立同人短篇小说自动化创作。

二、 竞品分析与核心优势

2.1 典型竞品拆解与对比

以市场上典型的网文辅助创作工具“星月写作”作为核心对标竞品，从产品机制、成本结构与合规风险三个客观维度进行对比拆解：

1\. 产品功能与运作机制对比

竞品机制（星月写作）： 采用高度分散的模块化UI设计，将工作流拆分为十余个独立的生成器入口（如书名、简介、大纲、细纲、金手指、人设等）。其核心商业化功能为内置的“AI扫榜”与“一键拆书”，系统可直接抓取并分析特定小说的核心爽点、金手指机制与人物设定。

本系统机制（差异点）： 商业化包装较弱，系统内部不集成任何网络爬虫或扫榜拆书模块，数据输入高度依赖用户自主将 .txt 格式的本地文件上传至特定的物理目录。其优势在于操作自主性更强，用户可对底层文件与工作流节点进行完全控制，不受单一平台的功能入口限制。

2\. 商业模式与成本结构对比

竞品劣势（商业溢价与受众冲突）： 星月写作采取高溢价的订阅制套餐模式（如白银包月79元，黄金包月119元）。由于其底层可能调用成本较高的商业大模型（如 Gemini），导致平台的刚性运行成本极高，因此对用户的上下文处理长度进行了严格的字数限制（如基础套餐单次关联知识库上限仅为3000字，超出部分需额外扣除字数额度）。这种高昂的定价策略与番茄小说等免费阅读平台的中底层作者（绝大部分月收入低于3000元，甚至不足1000元）的实际支付能力与降本需求产生严重冲突。

本系统优势（纯粹的 API 计费）： 本系统采用本地部署与云端 API 协同运作，用户直接对接底层大模型（如 DeepSeek）。系统运行产生的唯一支出为按 Token 实际消耗量结算的 API 基础调用费用。剥离了第三方平台的商业溢价，整体使用成本极低，完全契合低收入作者群体的成本敏感型需求。

3\. 合规要求与侵权风险对比

竞品风险： 星月写作等商业平台通过内置功能，直接在服务器端批量处理、解析并向用户分发受版权保护的热门商业网络文学设定数据，存在极高的平台级版权侵权与法律诉讼风险。

本系统优势： 作为一个纯本地化的辅助处理工具，本系统自身不包含、不提供、也不主动抓取任何外部版权文本。所有参考特征的数据源均要求用户自行存储至本地硬盘（reference\_novels 目录）。这种物理隔离机制将数据获取的行为主体转移至用户个人，基本规避了系统及开发者层面的直接侵权风险。

2.2 核心产品优势与护城河定位

系统的核心竞争力建立在以下三项底层机制之上：

第一项为分层检索结构（Hierarchical RAG）。系统通过本地算法（正则表达式切分与 Jieba 关键词提取）对前置小说正文进行章节维度的摘要降维。提取生成的摘要文本经向量化后存入向量数据库，并与其对应的长文本正文块建立双向映射关联。该机制能够在不额外消耗大语言模型 API 计费额度的情况下，有效释放模型的上下文处理窗口。

第二项为多版本树（Versions Tree）数据结构。在进行章节大纲续写或正文迭代时，系统将同一提示词指令生成的多个回答版本并行折叠于同一操作层级。该机制允许用户在智能对话区无损保留、比对并随时切换不同的剧情走向，为多分支剧情的推演提供物理存储支持。

第三项为知识库动态更新与强制读取机制。提取的角色信息卡与世界观架构具备版本更迭属性，系统在每次生成新章节后将重新提取设定，并采用附带章节锚点的方式（例如记录状态：第一章10岁、第五十章12岁）存储最新状态。系统提供可视化表单供用户执行手动增删改查，并在底层工作流中设置绝对优先级，强制大语言模型在生成阶段直接读取该面板最新落盘的物理数据，杜绝缓存调用带来的事实幻觉。

三、 产品功能与信息架构

3.1 界面布局与交互映射

系统前端界面物理划分为左、中、右三大固定区域，具体布局与组件映射如下：

左侧区域（全局控制台）：负责系统全局环境配置与当前对话状态管理。

布局与显示面板：包含右侧工作台界面的独立显示开关、中侧对话区与右侧编辑器宽度的占比调节滑块，以及控制系统全局文本的字号调节滑块。

API与防护面板：包含大语言模型 API 密钥输入配置框、全局禁用词拦截判定输入框。提供数值输入组件用于设定“自动重试次数”与“敏感词容忍度”，控制流式请求在触发限制条件时的自动重新发包逻辑。

历史对话管理面板：包含新建独立对话按钮、当前对话上下文设定一键无损复制（克隆）按钮。底部配备列表组件用于展示历史对话记录并提供点击切换功能。

当前对话参数面板：包含对话标题输入框、处理模型单选切换组件（DeepSeek V3 标准模型或 R1 推理模型）、当前对话专属的局部禁用词输入框。提供控制文本生成随机性的 Temperature 浮点参数滑块、控制词汇采样范围的 Top-P 浮点参数滑块、系统提示词（System Prompt）输入文本域，以及一键清空当前选定对话历史记录的执行按钮。

中侧区域（智能对话区）：负责执行大语言模型的指令收发及流式文本呈现。

顶部信息栏：静态展示当前激活的对话名称与运行的大模型标识，动态展示触发局部与全局禁用词规则的实时拦截统计整数次数。

消息展示区：以对话气泡形式渲染用户指令与大模型输出结果。在接收大模型流式数据流时，前端 JavaScript 实时执行正则表达式匹配，若捕获全局或局部禁用词，立即调用红底加粗 CSS 样式对违禁词进行高亮渲染。引入多版本树渲染逻辑支持同一层级的多版本切换。光标悬浮于正文消息框时触发“导入小说末尾”按钮，该按钮绑定后端追加写入 API。

底部输入区：包含多行文本输入框与发送请求按钮。监测到流式网络请求进行时，发送按钮物理切换为强制中断当前 TCP 请求的停止按钮。

右侧区域（小说工作台）：负责小说项目具体内容的执行编辑与物理文件管理。

内容编辑面板：顶部配备当前工作项目选择下拉菜单、特定章节选择下拉菜单、新建空白章节按钮与重命名当前文件按钮。核心区域为大尺寸纯文本正文编辑器，绑定实时字数统计计算事件，并内置防抖（Debounce）保存机制，在用户停止键盘敲击行为 1000 毫秒后自动触发后台文件覆写命令。

自动化工作流面板：上部为特征提取与统计区，提供参考小说源文件选择菜单与触发统计提取脚本的执行按钮。下部为项目初始化区，提供新项目书名输入框与一键生成标准工程结构按钮。

知识库管理标签页：将功能模块生成的非结构化或半结构化设定（关键词库、角色卡、世界观）转换为可交互的表单输入框，提供明确的编辑落盘通道。

3.2 核心业务工作流 (Workflow)

系统依托本地文件管理模块实现标准化的多级目录物理映射与数据流转。

底层目录架构：根目录下划分 reference\_novels（原文输入库）、text\_style\_imitation（特征与设定提取数据中台）、novel\_projects（具体小说创作工作台工程目录）、dictionaries（基础全局与停用词库）等子文件夹。

执行流：文本输入通过本地文件 I/O 直接产生对应目录下的 .txt 或 .md 文件更新。大语言模型交互枢纽调用 DeepSeek API 将生成的优质数据流推送至中侧智能对话区。用户进行有效性确认后，点击导入按钮，系统自动定位小说工作台目录下的目标项目章节文件，执行无损追加写入操作。当出现边界异常（未选中具体章节执行追加）时，API 保底路由算法自动读取所在项目目录，默认将文本追加至排序序号第一的物理文件末尾。

3.3 核心功能模块定义

系统后端业务逻辑脚本划分为七大核心功能模块：

功能一：特征提取

执行逻辑：分为本地物理指标计算与大模型深层文风提取两个阶段。系统采用异步进程模式拉起本地 NLP 算法脚本处理参考小说全量文件，计算长短句方差、高频词、标点使用频率及对话提示语位置分布比例。随后截取长度限制在 50000 字以内的原文切片发送至大模型，提取世界观标签、叙事节奏、语体色彩、人称视角、描写风格及修辞手法偏好。

进阶方向：构建文风偏移量化监测机制。对新生成的章节执行等效本地计算，提取指标并与原著基线数据计算数学距离（如欧氏距离），偏离预设阈值时自动拦截提示重试。

功能二：关键词库提取

执行逻辑：调用 Jieba 分词器处理全量文本，统计有效词与独立词汇总数。采用对数与根号混合动态推荐算法公式计算提取词汇名次上限（最高不超过 8000 词，支持用户手动输入整数名次强制提取前列高频词）。将全量高频词转换为词向量并在本地库中检索出关联的历史文本块，合并输入大语言模型执行词汇清洗、去重与实体分类，建立基础关键词库。

进阶方向：引入海量常规文本数据库比对，生成不包含在本项目高频范围内的负面词库。引入词频-逆文档频率（TF-IDF）与 TextRank 算法为词汇分配数学权重，在向模型发送指令时通过权重数值约束稀有专有名词的堆砌频率。

功能三：专属词库、世界观与角色信息提取

执行逻辑：取消原文单次处理的 50000 字物理截断限制，启用全局 1024 维向量检索。首先基于基础词库提取具体的人物名（含别名）、势力名、力量体系及地名建立专属词库。其次将核心实体词转换为检索条件输入全局向量数据库，召回全量文本中对应的高相关度段落。最后将召回段落提交大模型，强制输出具有统一格式要求的人物身份、境界、性格特征、人际关系与宏观世界观设定结构化文档。

进阶方向：建立角色信息卡的时间轴版本库。不再采用静态文档覆盖，而是通过章节编号生成锚点并建立追踪字典。向大模型传递设定时实行严格隔离，自动屏蔽旧版本状态并只发送最新锚点对应的数据。

功能四：设定补全与剧情压缩

执行逻辑：应对输入设定不全或上下文过长的两类状况。一方面，在大语言模型判定设定信息残缺时，结合现有词条推演补齐剩余的种族体系、境界等级或角色缺失属性。另一方面，利用正则表达式按章节符对前置正文执行切割，生成局部文本摘要并完成向量化，在此基础上与被切分的长文本原句建立关联，构建“摘要-正文”分层检索（Hierarchical RAG）索引库。

进阶方向：实现叙事结构模板的抽象提取。利用自然语言理解算法扫描原著剧情起伏，标注触发事件、冲突升级、高潮节点、结局收束四个核心阶段。在后续创作中将该结构化模型转化为硬性底层规则输入模型，规范新段落的物理发展框架。

功能五：小说创作

执行逻辑：构建在 RAG 检索机制与提示词分层约束上的长文本流式生成。用户输入基础剧情大纲，系统进行摘要向量化计算并从历史数据库中召回最高相似度前置文本块（含人物对话记录与环境描写细节）。将召回内容与提示词混合后请求模型生成详细版结构大纲，经用户确认后开始正文生成。同人分支调用原著设定与细节检索；原创分支仅调用自身积累的设定与已生成前文记录。

进阶方向：执行严密的上下文消息列（Message List）隔离方案。将被动约束信息（角色卡、底层逻辑等）填入 System Prompt，并附加强制性的否定指令（禁止在叙述中进行机械罗列与直接陈述），将主动输出信息（本章大纲、场景指引）填入 User Prompt，优化模型的指令执行依从度。

功能六：剧情方向推演

执行逻辑：扫描当前工作台已生成的所有正文与整体世界设定。系统自动提取“未解决事件”、“未使用道具”及“未确认死亡或下落的非主要角色”等休眠实体信息。将提取出的逻辑链条提交大模型，要求输出 3 至 5 个基于现存设定的合理后续发展分支报告。

进阶方向：开发强制休眠线索追踪算法。遍历早期前置摘要集，定向定位休眠实体并按照章节跨度距离进行权重计算（时间跨度越大权重越高）。将高权重线索作为强约束初始参数强制传入剧情推演模块，迫使大语言模型优先处理早期未收束的设定伏笔。

功能七：校验文本

执行逻辑：确保生成内容的客观逻辑闭环。系统提取待校验新文本中的人物动作、交互实体及数值变更参数。对这些核心要素进行特征向量化处理并在数据库内发起定向检索。调用大模型将新文本段落与提取出的项目基础设定、世界观规范、本章预设约束及 RAG 召回的对应历史事实块进行交叉比对。最终输出校验报告，定位并标示任何设定偏离、逻辑断层或数值错误问题。

4.2 算力分配与本地协同策略

系统采用“端-边-云”三层物理算力分配与协同调度机制，以保障高并发环境下的系统稳定性与经济性：

前端计算层（浏览器客户端）：承担流式文本渲染与正则匹配任务。大模型返回的文本数据流在到达前端后，直接由客户端 CPU 执行 JavaScript 正则表达式扫描，进行敏感词比对拦截。同时通过 marked.js 将 Markdown 标签实时转换为 HTML 富文本，降低服务器渲染压力。

本地后端计算层（物理服务器/个人电脑）：承担高密度文本处理与向量检索任务。系统底层使用 FastAPI 框架，针对耗时极长的全量小说文本分析（如 Jieba 分词、高频词统计、句段长度方差计算），采用 Python 的 subprocess.Popen 进程级异步调用模式，隔离主线程，防止阻塞前端 API 路由。本地硬盘直接承担 1024 维度的全局与分层向量数据库（RAG DB）的构建与检索开销。

云端计算层（大模型 API）：承担深层语义理解与文本生成任务。系统仅将截断后的原文样本（50000字内）或通过本地 RAG 精准召回的相关文本块发送至 DeepSeek V3（标准模型）与 R1（推理模型）的云端接口，执行深层文风提取、角色信息卡建立及分章正文生成。

4.3 异常处理与降级机制

系统在接口通信与文件读写层面设立了四套硬性防护与降级规则：

API 自动重试机制：系统控制台提供“自动重试次数”参数输入。当发送至 DeepSeek API 的网络请求遭遇 HTTP 500/502/504 错误、连接超时或被云端软阻断时，系统后端按预设整型数值执行线性重发，避免生成中断。

敏感词拦截与阻断策略：系统底层同时运行全局敏感词库与当前对话专属的局部禁用词库。在流式接收文本时，前端执行实时标红高亮。当单次生成的敏感词触发次数超过“敏感词容忍度”整型参数阈值时，前端即刻向后端发送中止指令，切断当前 TCP 传输连接，并视配置决定是否重新拉起生成请求。

文件编码自动降级机制：针对多源输入的纯文本（.txt）参考小说，底层文件 I/O 执行默认 UTF-8 编码读取。若系统捕获到 UnicodeDecodeError 异常，底层逻辑将自动无缝降级使用 GBK 编码进行二次文件流读取。

边界路由防丢机制：在执行大模型生成文本的保存操作时，若用户在右侧工作台未明确选中具体的目标章节文件即点击“导入小说末尾”，API 保底路由算法将自动扫描当前小说项目的物理目录，默认将文本数据追加至按照名称或创建时间排序第一的物理文件末尾，确保生成的数据实现物理落盘。

六、 问题复盘与演进规划

6.1 当前系统存在问题与边界局限

超大文本特征提取阻塞：本地 NLP 脚本在处理超过 300 万字级别的巨量网文原著时，即便采用异步进程，计算高频词对数 TTR 与断句方差仍需消耗大量本地 CPU 与内存资源，导致中低端 PC 设备出现短暂的系统卡顿。

模型注意力衰减（Lost in the Middle）：在执行功能五（小说创作）时，当 RAG 召回的碎片化文本块超过 10 个以上，且被统一打包至提示词中段时，DeepSeek V3 模型仍偶尔出现忽略部分约束设定的现象，导致生成的动作细节与补充设定的数值产生微小偏差。

系统级约束指令依从性波动：在将带有绝对否定指令（如“禁止出现某词汇”）放入 System Prompt 时，模型基座的底层对齐逻辑有时会覆盖系统的强制要求，导致负面词库拦截率无法达到物理层面的 100%。

6.2 后续产品功能迭代路线图 (Roadmap)

系统后续开发将围绕四个技术维度的“进阶方向”展开物理代码落地：

阶段一：文风偏移量化监测系统接入

对后续每次生成的新章节，实时执行等效本地 NLP 特征提取（如对话位置占比计算）。将其与原始参考文本的基线数据进行欧氏距离计算，当偏离值超出安全阈值时，自动拦截落盘并提示用户执行重试操作。

阶段二：动态词库权重与角色卡时间轴系统落地

引入 TF-IDF 与 TextRank 算法处理关键词库，向大模型发送带权重的专属词典，利用参数约束控制罕见专有名词的生成频率。重构角色信息卡物理文件格式，采用附加章节锚点的动态字典结构（如 年龄：\[第一章:10岁] -> \[第五十章:12岁]），在 RAG 检索时利用后端过滤算法强制屏蔽旧状态，仅输出最新锚点数据。

阶段三：叙事结构模板抽取算法研发

开发针对小说原著的自然语言分析分类器，标注出“触发事件”、“冲突升级”、“高潮节点”、“结局收束”核心单元。将抽取的抽象结构转化为底层规则参数，在功能五（小说创作）阶段约束新生成的章节在宏观走向上符合原始行文框架。

阶段四：休眠线索自动化追踪机制部署

完善功能六（剧情方向推演）。建立后端轮询脚本，遍历前置章节摘要数据集，自动定位已生成但未产生物理影响的实体（如未消耗的道具、未确认下落的角色）。按照章节跨度距离进行权重排序，强制转化为推演提示词的初始输入条件，干预模型的剧情分支树生成。
