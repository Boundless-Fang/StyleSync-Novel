# StyleSync-Novel 技术文档

## 1. 项目概述

StyleSync-Novel 是一个面向小说创作场景的自动化辅助系统，目标是在本地可控环境下，利用大语言模型与检索增强生成技术解决长文本创作中的三个核心问题：

- 角色设定冲突
- 上下文遗忘
- 文风不统一

项目当前主要服务两类场景：

- 同人小说衍生创作
- 原创小说连载辅助生成

从产品形态上看，它目前是一个本地可运行的 MVP 原型，而不是正式商用 SaaS。

## 2. 市场现状与需求洞察

### 2.1 行业背景

网络文学行业正在经历内容生产方式与商业模式的双重变化：

- 传统付费阅读与重度影视化改编模式增长放缓，平台需要寻找新的增长曲线
- 免费阅读平台扩大了用户规模，但作者收入分化明显，中腰部创作者的创作投入与回报不匹配
- AI 漫剧、微短剧和轻量化 IP 改编为网文带来新的变现路径
- 网文出海进一步强化了低成本、规模化、多语种内容生产的需求

在这个背景下，低成本、高可控、能提升生产效率的 AI 辅助创作工具具备现实需求。

### 2.2 目标用户画像

#### 用户一：原作者

- 核心需求：正文续写与连载辅助
- 关注点：API 成本、稳定性、上下文连贯性
- 特征：成本敏感，希望降低高频更新的负担

#### 用户二：同人创作者

- 核心需求：生成符合原著设定与风格的衍生内容
- 关注点：角色还原度、世界观保真度、风格一致性
- 特征：更关注内容是否“像原著”

### 2.3 核心痛点

- 角色设定冲突：模型容易违背既有的人物设定、关系结构和能力体系
- 上下文遗忘：长文本场景下，模型容易丢失早期伏笔和历史状态
- 文风漂移：不同轮次生成后，句式、节奏、词汇偏好容易断层

### 2.4 潜在延伸需求

- 基于原著设定的角色对话与互动
- 基于角色卡与世界观切片的自动化同人短篇创作

## 3. 竞品分析与项目差异

### 3.1 典型竞品特征

市场上的网文创作辅助工具通常具备这些特点：

- 采用平台化封装，流程入口固定
- 部分工具集成“扫书”“拆书”等重度分析功能
- 多采用订阅制，用户支付的是平台溢价后的服务价格
- 平台侧统一处理版权文本，存在更高的合规风险

### 3.2 本项目的差异化定位

StyleSync-Novel 选择了不同的路线：

- 本地主导：输入文本由用户自行管理，不内置采集和爬取功能
- 低平台溢价：主要成本来自底层 API 调用，不依赖高价订阅平台
- 高自主性：用户可以直接查看、编辑、维护本地文件和中间产物
- 风险更低：系统本身不主动抓取外部版权文本，数据输入主体是用户

### 3.3 核心优势

- 分层 RAG：通过“摘要索引 + 正文映射”释放大模型上下文窗口
- 多版本对话树：支持同一轮生成结果的并行保留和切换
- 知识库动态更新：角色卡和设定可视化维护，并参与后续生成约束

## 4. 产品目标与信息架构

### 4.1 产品目标

本项目希望验证两个问题：

- 在小说创作场景中，LLM + RAG 是否能显著改善长文本生成质量
- 本地工作流驱动的创作工具是否能比“单轮 prompt”更可控

### 4.2 前端布局

界面分为左、中、右三块：

- 左侧：全局控制台，包含模型参数、敏感词、历史对话管理
- 中侧：智能对话区，负责流式交互与版本切换
- 右侧：小说工作台，负责项目管理、章节编辑与自动化工作流

### 4.3 核心工作流

主流程可概括为：

1. 导入参考文本
2. 执行文本统计与风格提取
3. 生成关键词库、世界观、角色卡
4. 构建分层检索库
5. 生成章节大纲
6. 生成正文
7. 后续执行剧情推演与一致性校验

## 5. 技术架构

### 5.1 架构概览

系统采用前后端分离 + 本地文件持久化 + 多脚本工作流调度的形式：

- 前端：静态页面，负责交互与流式输出渲染
- 后端：FastAPI 路由层，负责接口编排和任务调度
- 本地计算层：负责文本处理、词频统计、FAISS 索引构建
- 云端模型层：负责深层语义理解与长文本生成

### 5.2 核心技术栈

- FastAPI：接口服务与工作流入口
- asyncio：任务调度与并发控制
- Jieba：本地中文分词与词性分析
- FAISS：本地向量数据库
- BAAI/bge-m3：向量编码模型
- DeepSeek 系列模型：风格提取、设定整理、正文生成
- 本地文件系统：项目结构、设定、章节和索引持久化

## 6. 关键模块设计

### 6.1 配置层

配置层已统一收口到 `_core_config.py`，主要负责：

- 路径常量维护
- `.env` 加载
- 默认模型管理
- API key 读取与别名兼容
- Hugging Face 缓存与环境初始化

### 6.2 LLM 调用层

当前采用轻量收口策略，没有完全重写，但已统一：

- DeepSeek 请求入口
- 通用重试状态码
- 聊天、流式输出、embedding 的 timeout 约定
- 公共请求头与 session 复用
- 错误处理风格

涉及文件：

- `api/routecore.py`
- `core/_core_llm.py`
- `core/_core_rag.py`

### 6.3 项目创建层

`routeproject.py` 中的 `create_project()` 已做同文件内轻量拆分，当前职责划分为：

- `build_project_name()`
- `init_project_structure()`
- `init_fanfic_project()`
- `init_original_project()`
- `ensure_required_files()`

这种重构方式保持接口行为不变，但提升了主流程可读性。

### 6.4 任务调度层

系统通过 `tasks.py` 管理后台任务：

- 记录任务状态
- 管理子进程执行
- 捕获 stdout/stderr
- 做基础 watchdog 监控
- 支持“终止最近任务”和“终止全部任务”

目前依然是原型级设计，后续如需继续产品化，可进一步拆分状态层和存储层。

## 7. 七个核心功能模块

### 功能一：风格特征提取

- 统计段落长度、句长分布、标点密度、词汇丰富度
- 提取叙事节奏、视角、描写方式和修辞倾向
- 输出风格约束文件

### 功能二：关键词库提取

- 基于 Jieba 分词和动态阈值筛选高频词
- 控制长文本下的词汇提取规模
- 生成基础关键词库

### 功能三：世界观与角色信息抽取

- 从全文中检索角色与实体相关上下文
- 生成结构化世界观与角色信息卡
- 为后续生成提供稳定约束

### 功能四：设定补全与剧情压缩

- 对不完整设定做结构补全
- 切分前文并生成局部摘要
- 建立摘要到原文的分层 RAG 结构

### 功能五：小说创作

- 输入剧情简述
- 召回相关前文与设定
- 先生成大纲，再流式生成正文
- 支持同人与原创两类工作流

### 功能六：剧情方向推演

- 提取未收束线索、未使用道具、休眠角色
- 基于既有设定推演多个后续剧情方向

当前属于规划阶段。

### 功能七：文本一致性校验

- 比对新文本与既有设定、世界观和前文事实
- 标记逻辑冲突、设定偏差、数值错误

当前属于规划阶段。

## 8. 模型策略与技术选型

### 8.1 选择原则

本项目选择模型时主要看五个维度：

- 中文文本质量
- 指令执行稳定性
- 输出长度
- 成本
- 合规和接入便利性

### 8.2 当前策略

当前主策略是：

- 标准生成任务：DeepSeek V3 类模型
- 推理和复杂判断任务：DeepSeek R1 类模型

主要原因：

- 成本更适合本项目目标用户
- 中文生成表现较稳定
- 更适合本地工具 + 云端 API 的工作模式

### 8.3 本地协同策略

采用“本地处理 + 云端生成”的分工：

- 本地负责：词频统计、文本切分、向量检索、文件管理
- 云端负责：风格总结、设定抽取、大纲生成、正文生成

这样可以减少 API 消耗，并保持流程可控。

## 9. 运行配置

### 9.1 最小环境要求

- Python 3.11 或 3.12
- 能访问相关模型 API

### 9.2 `.env` 最小配置

```env
DEEPSEEK_API_KEY=your_deepseek_key
SILICONFLOW_API_KEY=your_siliconflow_key
DEFAULT_CHAT_MODEL=deepseek-chat
DEFAULT_EMBEDDING_MODEL=BAAI/bge-m3
```

公开仓库中建议提交 `.env.example`，而不是 `.env`。  
使用时由测试者或开发者自行复制模板并填写真实密钥。

### 9.3 启动方式

```powershell
python style_imitation_code/main.py
```

默认监听：

- `http://127.0.0.1:8000`

### 10.3 当前测试边界

当前测试主要覆盖接口层与轻逻辑，不覆盖：

- 真实大模型生成质量
- 完整端到端内容生产链路
- 大体量向量库性能与压力测试

## 11. 当前限制与风险

### 11.1 工程限制

- 超长文本处理仍然会消耗较多本地 CPU 和内存
- 多阶段脚本链路依赖本地目录结构稳定
- 任务系统目前仍以本地 JSON 和内存状态为主

### 11.2 模型限制

- 超长上下文下仍可能出现注意力衰减
- System Prompt 的约束并非总能 100% 执行
- 文风稳定性受召回质量与提示词设计影响

### 11.3 产品限制

- 当前是本地原型，不是正式在线服务
- 没有面向普通用户的极简安装器
- 对外部测试者来说，环境搭建仍有一定门槛
