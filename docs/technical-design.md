# StyleSync-Novel 技术设计文档

## 1. 项目概述

StyleSync-Novel 是一个面向小说创作场景的本地 AI 原型系统，目标不是做成完整商业化 SaaS，而是验证以下能力是否能在真实创作流程中成立：

- `LLM + 风格分析 + RAG` 是否能提升长文本创作的一致性
- 工作流式创作工具是否比单轮 prompt 更可控
- 结构化生成与人工微调能否组合成更实用的创作体验

它主要服务两类场景：

- 同人创作：强调风格还原、设定一致与角色稳定
- 原创创作：强调设定管理、章节规划与持续续写

## 2. 产品目标

当前版本重点验证三件事：

1. 参考文本分析是否能为后续生成提供可复用的风格与设定资产
2. `f5a -> f5b` 的分层生成是否优于直接写正文
3. “工作台”是否能成为自动生成之外的人工干预层

因此，系统并不是围绕“聊天”组织，而是围绕“创作链路”组织。

## 3. 核心问题

项目试图缓解以下创作痛点：

- 角色设定容易漂移
- 长文本创作中上下文容易遗忘
- 多轮生成后文风不稳定
- 用户想改局部内容时，不希望每次都重跑全流程

## 4. 系统结构

整体采用“前端工作台 + FastAPI 接口层 + 本地脚本工作流 + 本地文件存储”的结构。

```text
Browser UI
  ├─ 左侧：系统设置 / 历史对话 / 任务队列
  ├─ 中间：工作台（聊天、微调、prompt 注入）
  └─ 右侧：内容编辑 / 自动化工作流 / 知识库管理

FastAPI API
  ├─ 项目与章节管理接口
  ├─ 工作流触发接口
  ├─ 对话与流式返回接口
  └─ 任务队列与状态接口

Script Layer
  ├─ 参考文本分析脚本
  ├─ 设定补全脚本
  ├─ 大纲生成脚本（f5a）
  ├─ 正文生成脚本（f5b）
  └─ 文本校验与扩展脚本

Storage
  ├─ reference_novels/
  ├─ text_style_imitation/
  ├─ novel_projects/
  └─ system_tasks_db.json
```

## 5. 技术选型

- FastAPI：接口服务与任务入口
- asyncio：异步任务调度
- 原生 HTML + JavaScript：前端原型实现
- Jieba：中文分词与基础统计
- FAISS：本地向量检索
- `BAAI/bge-m3`：向量嵌入模型
- DeepSeek 系列模型：设定提取、生成与分析任务
- 本地文件系统：项目、章节、知识库与中间结果持久化

## 7. 核心模块

### 7.1 配置层

负责：

- 路径常量管理
- `.env` 加载
- 默认模型读取
- API Key 兼容读取

### 7.2 LLM 封装层

负责统一处理：

- 聊天请求
- 流式输出
- embedding 请求
- 错误处理与超时控制

### 7.3 项目管理层

负责：

- 项目创建
- 章节创建与切换
- 正文内容读写
- 知识库文件读写

### 7.4 任务调度层

负责：

- 任务状态记录
- 子进程执行
- stdout / stderr 采集
- watchdog 监控
- 终止最近 / 全部 / 勾选任务
- 清除最远 / 全部 / 勾选记录

当前设计上，任务层只负责“运行与记录”，还没有实现完整的“终止即回滚全部半成品”机制。

## 8. 主工作流

当前主工作流可以概括为：

1. 导入参考文本
2. 执行风格分析与词汇提取
3. 生成世界观、角色卡等知识资产
4. 构建项目级上下文与记忆
5. 使用 `f5a` 生成章节大纲
6. 使用 `f5b` 生成章节正文
7. 使用工作台进行局部微调、prompt 改写和人工干预

## 9. `f5a` 设计

### 9.1 定位

`f5a` 是章节大纲生成节点，解决“这一章应该怎么写”的问题。

它位于设定补全之后、正文生成之前，属于规划层。

### 9.2 输入

当前前端可组织的输入包括：

- 章节名
- 章节概述
- 情节阶段
- 小说阶段
- 章节功能
- 叙事人称与视角
- 角色列表
- 目标字数
- 节奏与场景切换
- 开头 / 铺垫 / 高潮 / 收束四段结构信息

### 9.3 输出

`f5a` 产出结构化章节大纲，作为后续 `f5b` 的上游资产，也可在知识库或工作台中继续人工改写。

### 9.4 产品意义

`f5a` 的价值不是单纯“多一个脚本”，而是把“先规划后写作”显式产品化：

- 方便展示结构化创作思路
- 方便解释章节控制逻辑
- 为 `f5b` 提供更稳定的写作约束

## 10. `f5b` 设计

### 10.1 定位

`f5b` 是正文生成节点，负责把项目设定、章节信息与上下文转换成小说正文。

它位于主工作流的执行层，是最接近最终创作结果的节点。

### 10.2 输入

`f5b` 当前主要依赖：

- 当前项目
- 章节名
- 项目中的设定与知识资产
- 章节大纲与相关上下文
- 选定模型

### 10.3 输出

生成结果会写入小说内容编辑区，并作为当前章节正文展示。

### 10.4 Prompt 注入能力

这是当前版本相对更有辨识度的设计。

在 `f5b` 执行区，用户可以选择：

- 执行脚本后把 `f5b` 提示词注入工作台输入框
- 进一步选择“不生成小说”

这形成了三种模式：

1. 正常生成：直接执行 `f5b`
2. 注入后继续生成：先把本次 prompt 注入工作台，再继续生成
3. 只导出 prompt：注入工作台，但不执行正文生成

这让 `f5b` 从“黑盒生成器”变成了“可观察、可导出、可干预”的生成节点。

## 11. 工作台设计

### 11.1 定位

工作台位于中间区域，是当前界面中承接人工干预的核心层。

它不是通用聊天区，而是面向小说创作的微调控制台。

### 11.2 当前能力

快捷模板：

- 修改角色卡
- 修改世界观
- 润色章节大纲
- 生成正文
- 修改小说正文

注入来源：

- 知识库内容
- 小说正文内容
- `f5b` 导出的提示词

### 11.3 设计目的

工作台解决的是“结构化流程很好，但用户仍然想对局部做即时干预”的问题。

它的作用包括：

- 作为 prompt 试验区
- 作为知识库内容的二次加工区
- 作为正文修订区
- 作为观察 `f5b` 生成逻辑的入口

### 11.4 与知识库的关系

知识库负责存储结构化创作资产，工作台负责调用这些资产进行动态操作。

典型链路：

1. 打开知识库文件
2. 点击“注入工作台”
3. 工作台自动填入模板与内容
4. 继续修改、追问或发送

### 11.5 与正文编辑区的关系

正文编辑区也可将当前章节内容注入工作台，用于：

- 润色
- 改写
- 修补逻辑
- 保留剧情事实前提下的局部优化

## 12. 前端信息架构

### 12.1 左侧

- 布局与显示设置
- 历史对话与新建对话
- 当前对话参数
- 任务队列

### 12.2 中间

- 工作台消息区
- 快捷模板区
- 输入框与动态高度调节

### 12.3 右侧

- 内容编辑
- 自动化工作流
- 知识库管理

这种布局把“系统设置”“人工微调”“结构化资产与工作流”拆到了三个区域，减少了纯聊天式产品的歧义。

## 13. 任务队列设计

任务队列当前有两个动作组：

- `终止`
  - 终止最近
  - 终止全部
  - 终止勾选
- `清除`
  - 清除最远
  - 清除全部
  - 清除勾选

其真实语义如下：

- `终止`：停止后台进程，任务状态标记为 `cancelled`
- `清除`：移除任务队列中的记录

当前限制：

- `终止` 不是事务级回滚
- 不保证自动清理所有已写入的中间文件或半成品
- 如果后续继续工程化，可改造成“临时文件 -> 成功提交 -> 终止清理”的机制

## 14. 测试策略

当前测试以“最小必要回归”为主，重点覆盖：

- 接口参数校验
- 任务接口返回结构
- 项目创建与章节读写
- 聊天接口基础行为

现有测试文件：

- `tests/test_chat_api.py`
- `tests/test_workflow_api.py`
- `tests/test_project_api.py`

尚未系统覆盖：

- 真实生成质量评估
- 完整端到端创作链路评估
- 工作台注入后的人工微调效果评估

## 15. 当前限制

工程限制：

- 仍以本地 JSON 与文件系统为主
- 任务系统偏 MVP 实现
- 部分工作流节点更适合展示原型，而非生产部署

模型限制：

- 长上下文下仍可能出现注意力衰减
- 提示词无法保证百分之百执行
- 风格稳定性受召回质量和 prompt 设计影响

产品限制：

- 当前是本地原型，不是正式线上服务
- 外部测试者仍需自行配置环境
- `终止` 与“彻底清理半成品”之间仍有语义差距

## 16. 后续路线

如果继续迭代，优先级较高的方向包括：

- `f5a/f5b` 结果质量评估体系
- 工作台与知识库更深的联动
- 重点节点的安全边界与提示词模式切换
- 终止任务后的临时产物清理机制
- 文风漂移量化与一致性校验

## 17. 对求职展示的价值

这套设计最适合作为 AI 产品经理 / AI 应用方向作品展示的原因在于：

- 有明确工作流，而不是空泛概念
- 有真实前后端原型，而不是只有静态原型图
- `f5a` 和 `f5b` 分工清晰，便于讲产品拆解
- 工作台体现了“结构化自动化 + 人工干预”的产品思维
